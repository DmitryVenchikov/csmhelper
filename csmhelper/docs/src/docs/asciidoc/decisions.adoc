=== Архитектурные решения

==== ADR-1. Хранение пароля Jira в Session ASP.NET Core

Сервис не имеет своей пользовательской БД и не реализует хранилище секретов. Чтобы делать запросы к Jira от имени пользователя через Basic-auth, его пароль помещается в `Session` сразу после успешной аутентификации (`JiraService.AuthenticateAsync`).

Последствия:

* пароль доступен на стороне сервера всё время жизни Session-cookie (`IdleTimeout = 30 минут`);
* при перезапуске процесса все сессии теряются — пользователь должен войти заново;
* в логах пароль не выводится (см. `logging.adoc`).

Альтернативы, рассмотренные при проектировании:

* персональные API-токены Jira — отброшены: на корпоративном экземпляре это потребует отдельной процедуры выпуска и обмена;
* SAML/OAuth — отброшены: нет инфраструктурной поддержки в текущем экземпляре Jira.

==== ADR-2. Двойной флаг аутентификации `IsAuthenticated`

Помимо `JiraUsername`/`JiraPassword` в Session пишется отдельный ключ `IsAuthenticated = "true"`. Все защищённые endpoint'ы проверяют *оба* источника:

[source,csharp]
----
private bool IsAuthenticated() => HttpContext.Session.GetString("IsAuthenticated") == "true";
----

И затем, при переходе на страницу, дополнительно вызывают `IJiraService.IsAuthenticatedAsync()` (проверка наличия логина/пароля). Это страхует от рассинхрона между Session-ключами при ручном `Logout` или ручной чистке сессии.

==== ADR-3. Кастомная валидация SSL для `alfaintra`

В `Program.cs` `HttpClient` для Jira настраивается с кастомным `ServerCertificateCustomValidationCallback`, доверяющим внутренним сертификатам `alfaintra` (ошибка цепочки `RemoteCertificateChainErrors` игнорируется). Это нужно, потому что цепочка корпоративного CA может быть не установлена в контейнере.

В `GantService.FetchJiraTasksAsync` сделан более жёсткий обход: при `request.VerifySsl == false` валидация SSL отключается полностью. Решение принято осознанно: сервис деплоится в закрытый контур, и DPI/MITM здесь невозможны.

==== ADR-4. Все ответы — `200 OK` с `Success` в теле

Большинство POST-методов сервиса (`Auth/Login`, `Auth/Logout`, `Jira/CreateTasks`, `Cleanup/SearchTasks`, `Cleanup/ProcessTasks`, `Gant/Generate`) возвращают HTTP-код `200 OK` независимо от исхода — реальный статус лежит в JSON-поле `Success` либо `success`. Это упрощает фронтенд (один и тот же путь обработки), но затрудняет «грубый» мониторинг по HTTP-кодам. Альтернативный подход (4xx/5xx для ошибок) был отвергнут из-за исторических соглашений с UI.

==== ADR-5. Парсинг типа задачи по подстроке `summary`

В `Gant` тип задачи (`TEST` / `DEV BACK` / `DEV FRONT` / `SA`) определяется по подстроке в `summary` Jira (`DetectTaskType`). Это сделано, потому что в корпоративном экземпляре нет отдельного `customfield` под тип CSM-активности, а заводить такой с центральным админом дороже, чем договориться о соглашении на префиксы. Последствие: задачи без префикса в `summary` *молча игнорируются* при построении диаграммы Ганта.

==== ADR-6. Story Points через `customfield_10372`

Эталоном Story Points является `customfield_10372`. Поле читается «толстым» способом (поддержка `double`, `string`, `long`, `int`) с фильтром `> 0`, потому что Jira возвращает SP в разных типах в зависимости от версии плагина. Если SP отсутствует или `<= 0` — задача исключается из планирования.

==== ADR-7. Однопроходный list-scheduling с проходами

В `ScheduleBuilder.Build` реализован простой list-scheduling: задачи сортируются по числу блокеров и приоритету, на каждом проходе ставятся в расписание те, у которых все блокеры уже запланированы. На реальных объёмах CSM-плана (≤ 200 задач) этого достаточно. Альтернативный CPM-обход (forward/backward pass с `early/late start`) рассмотрен, но сейчас не нужен — критический путь считается отдельно после планирования.

==== ADR-8. Параллелизм при обработке Cleanup (батч по 3, лаг 500 мс)

`JiraService.ProcessTasksAsync` дробит задачи на батчи по `3` и параллелит их через `Task.WhenAll`, а между батчами держит паузу `500 мс`. Это эмпирически подобранный компромисс: меньшее число параллельных запросов и пауза избавляют от 429/перегруза Jira; больший батч раньше упирается в SLA Jira REST. Менять только после нагрузочной проверки.
